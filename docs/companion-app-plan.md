# Trafalgar companion app — plan

Goal: a phone/tablet companion. **Remote control now, standalone-native later.**

**Chosen topology: a hub.** One always-on hub process owns the phone-facing
surface and routes control to many plugin instances by name (e.g. `lead-synth`,
`drums`) — because embedding a listener in each plugin instance does not compose
(N instances would fight for one port, and the phone can't tell them apart). See
"Hub architecture" below. The phone→hub link can be any of the transports below;
the instance↔hub link is a small outbound OSC protocol.

Phone→hub transport options (all end up writing `shared.pos`/`shared.gate` in the
target instance — the audio thread can't tell a finger from a phone):

| Option | New deps | App to install | Notes |
|--------|----------|----------------|-------|
| **TouchOSC** (recommended) | none — `rosc` already a dep | TouchOSC ($) | Hub is pure OSC relay; no web server. Build the layout in TouchOSC's editor. |
| Web page + WebSocket | `axum`, `tokio` (hub only) | none (browser) | Hub serves a page with an instance picker. Like `../rust/rustjay-engine`. |
| MIDI in (Network/BLE) | none | any MIDI app | Best for hardware controllers; addressing many instances is awkward over MIDI. |

## Guiding decision

The plugin already routes every live gesture through two per-track shared atomics:

- `shared.pos[tr]` — packed X (pitch) + Y (density), written *before* the gate opens
- `shared.gate[tr]` — "pad is being touched"

A remote controller is just *a second thing that touches the pad*. Incoming MIDI
writes the same `pos` + `gate` the `XyPad` writes. No new engine concepts, no
param plumbing — the audio thread can't tell a finger from a phone.

So the whole roadmap is: feed that path from a remote now; run the engine on-device
later. The only question is *how the bytes reach the right instance* — the hub.

## v1 — BUILT: per-instance OSC input (no hub)

Shipped. Each instance can listen for pad control on a user-assigned port (the port
is the instance's identity — no hub, no names). Decided via `/grilling`; hub kept
below as the upgrade path.

**Settings popout → "OSC input (remote pad)":**
- `Enabled` — opt-in, off by default.
- `Port` — editable, default 9100; assign 9101/9102/… per instance.
- `LAN access` — on → binds `0.0.0.0` (phone reachable); off → `127.0.0.1` only.
- Status line: `off` / `listening` (green) / `port in use — OSC-in off` (red).

**Address contract** (send here from TouchOSC; `n` = 0..3):
```
/track/{n}/xy     ,ff  x y   # x = pitch 0→1, y = density 0→1
/track/{n}/gate   ,f   0|1   # ≥0.5 opens the pad gate, <0.5 closes
/track/{n}/erase  ,f   0|1   # momentary erase while held
/track/{n}/clear  ,f   1     # ≥0.5 wipes the track's recorded loop
```

**TouchOSC layout (shipped):** `docs/trafalgar.tosc` — load it in the TouchOSC app.
4 track columns, each an XY pad (its touch sends both `/xy` and `/gate`) plus Erase
and Clear buttons. Source: `docs/trafalgar.tosc.xml` (readable) regenerated by
`docs/trafalgar-tosc.gen.py` (edit colours/layout there, rerun `python3` to rebuild).

**Connect it:** in TouchOSC → *Connections → OSC*, set **Connection 1** to the
instance's `<host-ip>:<port>` (the port from the popup, e.g. 9101), Send on. The
layout sends everything on connection 1. To drive a *second* instance at once, open
the layout's controls and switch them to Connection 2, pointing at the other port
(the hub upgrade path removes this manual step). Arm **Record** on the desktop and a
phone-driven pad records the gesture — the remote gate opens the same `touching` path.

Untested caveat: the file is built to the reverse-engineered `.tosc` schema (zlib
XML, matched against TouchOSC-authored examples) and validated structurally, but it
has not been opened in the TouchOSC app here — if it won't load, the address contract
above lets you rebuild it by hand in a few minutes.

Notes (all `ponytail:`-marked in code): writes share the mouse's atomics, so
concurrent local+remote on the *same* track is undefined; phone erase/clear aren't
undoable (undo stack is GUI-only); a gate packet arriving before its first xy can
cost one stale step. Input is defensively parsed (unknown/malformed/out-of-range/
NaN dropped) since the socket is open on the LAN.

## Hub architecture (upgrade path, not built)

Why not a listener per plugin instance: each instance would `bind()` the same port
(collision), and the phone has no way to address "the drum one" vs "the lead one".

The fix leans on two facts already true in the codebase: instances **already send
OSC outbound**, and a UDP socket is **bidirectional**. So keep instances as clients.

```
  TouchOSC / web ──► HUB (one process, well-known port 9000)
                      │   name → source-addr map (from hellos, expires on silence)
                      ├──► lead-synth  (instance's ephemeral UDP source port)
                      └──► drums       (instance's ephemeral UDP source port)
```

- **Instance** (extend `osc.rs`): one UDP socket `connect(hub)`. On init and every
  ~2s send `/trafalgar/hello ,s <instance-name>` (heartbeat). A reader thread on the
  *same* socket receives control — `/track/{n}/xy ,ff` and `/track/{n}/gate ,f` —
  and writes `shared.pos[n]` / `shared.gate[n]` (the exact pack from `XyPad::apply`).
  No `bind` to a fixed port, so any number of instances coexist. Instance name + hub
  address live in the settings popout (persisted).
- **Hub** (new bin `trafalgar-hub`): `bind` UDP 9000. On `hello`, record
  `name → src_addr` + last-seen. On a control message `/trafalgar/<name>/track/...`
  from the phone, strip `<name>`, look up its addr, and forward `/track/...` there.
  Expire names not seen in ~5s. Pure `rosc` — no web server for the TouchOSC path.
- **Phone** (TouchOSC): one tab per instance; XY pads send
  `/trafalgar/<name>/track/{n}/xy` and their touch sends `/trafalgar/<name>/track/{n}/gate`.

This composes to any instance count, needs no discovery service (hellos self-register),
and reuses `rosc` on both ends. A web surface is a later swap-in on the hub only.

### Hub MVP slice (only if v1's manual ports/connections become annoying)
1. `trafalgar-hub` bin: bind 9000, keep a `HashMap<String, (SocketAddr, Instant)>`,
   handle `hello`, relay `/trafalgar/<name>/...` → `<name>`'s addr as `/...`.
2. Plugin: add `instance_name` + `hub_host`/`hub_port` to the settings popout
   (persisted); a control client that sends `hello`/heartbeat and receives xy/gate.
3. TouchOSC: one XY pad → `/trafalgar/lead-synth/track/0/xy`, touch → the gate addr.

Success = TouchOSC pad drives the instance named `lead-synth`, and a second instance
named `drums` ignores it. Proves registration, routing, and the shared-path reuse.

## Phases (single-instance / standalone paths, still valid)

### Phase 1a — TouchOSC + OSC input (recommended, do first)
Add an OSC *listener* (Trafalgar currently only sends). Reuses `rosc`; no new deps.

- New receive socket bound to e.g. `0.0.0.0:9001`, on a background thread (mirror
  `osc.rs`'s send thread). Decode with `rosc::decoder`, match the address, write
  `shared.pos[track]` then `shared.gate[track]` — the exact pack from `XyPad::apply`.
- Wire OSC-in host/port into the existing settings popout next to the OSC-out fields.
- OSC address contract:
  - `/trafalgar/track/{n}/xy  f f`  → x, y in 0..1 → pack into `pos[n]`
  - `/trafalgar/track/{n}/gate  f`  → ≥0.5 opens `gate[n]`, else closes
  - (TouchOSC XY pads send the first; set the pad's touch to send the gate.)
- The "control page" = a TouchOSC layout: 4 XY pads + 4 gate toggles pointed at
  those addresses. Built in the TouchOSC editor (a few minutes); ship the `.tosc`
  in `docs/` once made. `ponytail:` no app code, no server — off-the-shelf app.

Result: run TouchOSC on the phone, drag a pad, notes fire on that track.

### Phase 1c — Web remote (alternative to 1a)
The standalone serves a multi-touch page; touches drive the existing live path.

- New `web` module in the crate, started from `initialize()` (standalone only):
  spawn a tokio runtime + `axum` server (mirror `rustjay-control/src/web/mod.rs`):
  ```
  Router::new()
      .route("/", get(|| async { Html(include_str!("pad.html")) }))
      .route("/ws", get(ws_handler))     // ws.on_upgrade(handle_socket)
  // axum::serve(TcpListener::bind(("0.0.0.0", port)), app)
  ```
- `pad.html` = 4 multi-touch XY pads (pointer events, `touch-action: none`), one
  WebSocket. On touch it sends `{track, x, y, gate}`; the handler writes
  `shared.pos[track]` then `shared.gate[track]` — *the exact bytes `XyPad::apply`
  writes*. The audio thread can't tell a finger from a phone.
- Pass `Arc<Shared>` into the server; that's all it needs (plus params later for
  reading state back to draw playheads).
- Deps: `axum` (features `ws`), `tokio` (rt + net). New deps, but this is the whole
  feature — `ponytail:` justified.

Result: open the URL on a phone, drag a pad, notes fire on that track. No install.

### Phase 1b — MIDI input (for hardware, after 1a)
Same shared-path trick, different source. Enable MIDI **input** and map it on.

- `MIDI_INPUT = MidiConfig::Basic` (currently `None`).
- In `process()`, handle `NoteEvent::MidiCC` / `NoteOn` / `NoteOff` per the CC
  contract below, writing the same `pos` / `gate`.
- No custom app needed: TouchOSC / MIDI Designer already send Network/BLE MIDI.

### Phase 2 — Extract `trafalgar-core` (enabler for standalone)
Move the host-independent engine into its own crate so both desktop and mobile
share one sequencer:

- Pure logic already isolated and testable: `euclid`, `rotated`, `xorshift`,
  `scale_note`, `Scale`, the gesture buffer, and the per-step generation in
  `process()`. None of it needs `nih_plug` except `NoteEvent`/transport types.
- New `crates/trafalgar-core`: owns state + a `fn tick(step, inputs) -> Vec<NoteOut>`.
  Desktop plugin and mobile app both call it. Keeps the existing tests.
- Do this only when starting Phase 3 — YAGNI until then.

### Phase 3 — Standalone native
Rust core (`trafalgar-core`) via **uniffi** bindings, thin native shells:

- iOS: SwiftUI UI, CoreMIDI / AudioKit out.
- Android: Jetpack Compose UI, `android.media.midi` / Oboe out.
- Alternative if two UIs is too much: one Rust `egui`+`wgpu` UI rendered on both
  (same pad code as a future desktop-standalone) — decide when we get here.

## MIDI CC contract (Phase 1b)

| Channel | Track      |
|---------|------------|
| 1–4     | track 0–3  |

| Message            | Meaning                                   |
|--------------------|-------------------------------------------|
| Note On (any note) | gate open (start touching this track's pad) |
| Note Off           | gate close                                |
| CC 16              | X → pitch (0..127 → 0..1)                  |
| CC 17              | Y → density (0..127 → 0..1)               |
| CC 18 (optional)   | base note                                 |
| CC 19 (optional)   | record on/off (≥64 = on)                  |

Mirrors `XyPad`: on Note On write `pos` then `gate`; CC updates `pos` live.
